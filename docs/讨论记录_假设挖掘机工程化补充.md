# 讨论记录：假设挖掘机的工程化补充

> 本文档记录围绕“假设挖掘机”的一轮工程化讨论。  
> 目标不是形成最终方案，而是保存有价值的中间判断，避免散落在聊天里遗忘。

## 一、核心转向：从“剪枝”改成“分级加工”

最初设想是：假设挖掘机会递归追问“这个观点背后的假设是什么”，然后用探针对可疑节点继续深挖，对不可疑节点剪掉。

这个方案后来被修正。

原因是：**不挖下去，就不知道哪一层有问题。**

一个看起来很普通、很无聊的浅层假设，可能在第 100 层、第 1000 层，甚至更深处才暴露真正的问题。如果过早剪枝，就会不可逆地丢掉信息。

因此正确设计不是“剪掉”，而是：

> **只排序，不删除。**

建议采用多级加工流水线：

1. **初加工**：快速扫面，广度优先，把观点或文本里的候选假设先挖出来，形成假设树。
2. **正加工**：给每个节点打可疑度分数，优先处理高分节点，但低分节点仍保留在队列中。
3. **深加工**：对高价值分支持续递归深挖，同时在每一层都重新运行探针。

一句话：

> **砍掉是赌博，排序是策略。**

## 二、为什么机器有价值：规模定律也适用于知识挖掘

人类很难长期追问到底，因为人的注意力、时间、精力都是强约束资源。

一个人可能追问 3 层、5 层、10 层就累了；但真正的裂缝可能藏在很深的地方。

这意味着：

> 地基裂缝之所以罕见，不一定是因为人类不聪明，而可能是因为没有足够资源把假设树挖得足够深。

大模型和自动化系统的价值，不只是“更聪明”，而是：

> **不知疲倦地执行深度搜索。**

规模定律不仅适用于大模型训练，也可能适用于人类知识的积累与审查：扫描得越广、追问得越深、案例积累越多，撞见关键裂缝的概率就越高。

## 三、边界标注器：不要说“触底就停”

曾提出一个“触底检测器”：当递归追问撞到物理极限、资源极限、数学公理时就停止。

这个方向有价值，但“停止”这个动作容易过早。

因为很多历史突破恰恰发生在：

> 大家以为撞到了物理极限，其实只是撞到了当时测量手段、工程条件或表达方式的极限。

因此更稳妥的叫法是：

> **边界标注器。**

它不是刹车，而是给当前节点标注“地层类型”：

* 这是物理边界吗？
* 这是资源边界吗？
* 这是制度边界吗？
* 这是表达边界吗？
* 这是公理边界吗？
* 还是只是“被误认为边界”的边界？

这样做可以避免把暂时的技术限制误判成真正的终点。

## 四、异常收集器：不要只从观点开始挖

单纯对所有观点递归追问，会很快发散。

更稳的入口是：

> **先从异常开始。**

异常是矿脉露头。真正的突破往往不是从“一个观点”开始，而是从“某个地方不对劲”开始。

因此假设挖掘机前面可以加一个模块：

> **异常收集器。**

它的任务是从论文、RFC、GitHub Issue、实验报告、标准文档中提取异常信号，例如：

* unexpectedly
* surprisingly
* however
* fails to
* does not generalize
* workaround
* compatibility issue
* edge case
* known limitation

更一般地说，要寻找这些模式：

* 多篇论文都遇到同一类失败。
* 多个 Issue 都在绕同一个限制。
* 一个标准为了兼容历史包袱反复引入补丁。
* 一个模型或理论在边界条件下反复失效。

新的流水线可以变成：

> **异常 -> 反推假设 -> 递归挖掘 -> 排序 -> 证据闸门 -> 条件与边界**

一句话：

> **不要从观点开始挖，要从异常开始挖；观点太多，异常才是矿脉露头。**

## 五、高价值矿区：论文、RFC、Issue、标准演进史

论文是重要来源，因为它们是领域内研究者的心血，知识密度高，假设、推理、证据链都高度压缩。

但论文也有局限：它往往是已经成功说服同行的文本，因此最深的隐性前提，可能正是整个同行圈共同默认的东西。

因此，除了论文，还应重视工程类材料：

* RFC 标准文档
* API 演进史
* GitHub Issue
* 设计讨论
* 兼容性说明
* 标准委员会记录

这些材料的价值在于：工程师往往会非常诚实地记录当年的限制与妥协。

例如：

> 因为当时带宽、内存、字段长度、兼容性或硬件条件受限，所以做出了某个设计选择。

多年后，原始限制可能已经消失，但设计选择仍然保留，并引发新的补丁、兼容成本和边界问题。

这类材料很适合“妥协遗忘探针”和“条件与边界”分析。

## 六、排序信号：哪些节点应该优先深挖？

由于不能删除分支，只能排序，因此需要设计可疑度评分。

可考虑以下维度：

### 1. 补丁密度

某个假设上方挂着多少补丁？

补丁越多，说明这个假设可能越可疑。

例子：

* 地心说后期的本轮与均轮。
* 软件工程里的格式化工具、语义合并、兼容补丁。
* AI 里的 prompt trick、RAG 补丁、工具外挂。

### 2. 年龄与跨度

这个假设存在了多久？横跨了多少子领域？

存在越久、覆盖越广，一旦出问题，价值越大。

### 3. 资源解绑度

这个假设最初依赖的资源限制今天是否已经消失？

例如：

* 早期显示器限制是否仍存在？
* 胶片成本是否仍存在？
* 机械卡键是否仍存在？
* 内存/带宽/字段长度限制是否仍存在？

如果原始限制已消失，而标准仍保留，就值得优先检查。

### 4. 异常聚集度

是否有大量异常都指向同一处？

单个异常可能只是噪音；多个异常反复撞到同一边界，才更像地基裂缝。

## 七、第一版原型不要做大

第一版不应该做成“大而全的人类知识扫描器”。

建议做一个很小的原型：

> 输入一篇论文或一个 GitHub Issue，输出四栏：
>
> 1. 异常句子
> 2. 可能前提
> 3. 可疑度评分
> 4. 下一层追问

示例：

```markdown
## 异常
模型在跨域数据上性能显著下降。

## 可能前提
训练分布与测试分布足够接近。

## 可疑度
高。该前提经常被默认，但在真实部署中反复失效。

## 下一层追问
这个“足够接近”依赖哪些条件？这些条件在真实场景中是否存在？
```

这个原型很小，但已经能体现项目价值。

它不是替研究者完成发现，而是帮研究者把“哪里不对劲”转化为“可以继续追问的假设树”。

## 八、当前版本的流水线草图

```text
输入：论文 / RFC / GitHub Issue / 标准文档 / 实验报告

1. 异常收集器
   提取“不对劲”的句子、失败、例外、workaround、edge case。

2. 假设挖掘机
   对异常反推前提，并递归生成假设树。

3. 分级加工器
   初加工快速扫面，正加工排序，深加工持续递归。
   只排序，不删除。

4. 探针评分器
   用补丁密度、年龄跨度、资源解绑度、异常聚集度打分。

5. 边界标注器
   标出物理边界、资源边界、表达边界、制度边界、公理边界等。

6. 证据闸门
   防止把普通裂纹夸成地震。

输出：候选隐性前提 / 成立条件 / 失效边界 / 下一步追问
```

## 九、最重要的工程原则

1. **不删除，只排序。**
2. **从异常开始，不从所有观点开始。**
3. **机器负责广度与深度，人负责判断与证据。**
4. **边界标注器不是刹车，而是地层标注。**
5. **原型要小，先证明一个最短闭环。**

一句话总结：

> 假设挖掘机不是一台“自动发现真理”的机器，而是一台把异常转化为可追问假设树的机器。

## 十、第一性原理视角：机器的根本价值是“对抗人类能量优化机制”

把所有讨论收束到最底层物理真相：

> 人类大脑是能量受限的物理系统（安静时约 20W，专注思考时约 35–40W）。智能的本质，是在能量、时间、注意力约束下的效率优化。

在这种物理约束下，任何一个智能系统都会演化出“剪枝机制”——把看起来不重要的追问、异常、假设提前砍掉，以节省能量。

因此：

> 人类不是“偶尔”会忽略地基，而是“必然”会忽略地基。

长期深挖假设树，在进化上是极度浪费能量的行为。所以人类会本能地对“司空见惯”的事情停止追问——这不是认知缺陷，而是**能量效率的必然结果**。

在此前提下，机器的真正价值不是“比人更聪明”，而是：

> **不受人类生物能量约束，可以不计成本地长期深挖假设树。**

这把“找到地基”和“对抗人类能量优化机制”统一成了同一个目标，而不是非此即彼。

设计上的直接推论是：

> **假设挖掘机的核心目标，应该最大化“深度搜索的持续性”，而不是最大化“单次发现的准确性”。**

这意味着：
- 不要急着让机器“聪明地剪枝”。
- 反而应该故意保留大量看起来很蠢、很低优先级的分支。
- 可以设计“故意浪费算力”的机制，定期随机深挖低分分支，防止人类式的效率捷径污染搜索过程。

在这个视角下，项目最终构建的，不是一个“更好的发现工具”，而是一个**人类认知的外部义肢（external cognitive prosthesis）**——它存在的意义，不是替人思考，而是**防止人因为能量限制而过早停止思考**。

一句话：

> 机器在这里不是在替人类变得更聪明，而是在替人类对抗自身能量优化机制带来的必然盲视。
