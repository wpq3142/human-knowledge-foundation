# 案例拆解 03：软件工程的“代码即纯文本”预设

**所属探针类型**：【地图误认探针】 (The "Map vs. Territory" Probe)

## 1. 当时默认了什么？（隐性前提）
从 1970 年代的电传打字机时代开始，我们就默认：**源代码的本体，就是一串保存在 `.py` 或 `.java` 文件里的英文字符。** “代码 = 文本文件”，这似乎是天经地义的物理定律。

## 2. 为什么这个默认在当时很合理？（当年的资源限制）
在早期的计算机系统中，存储空间极小，显示设备（如电传打字机或早期的 CRT 终端）只能逐行输出 ASCII 字符。将复杂的逻辑结构“降维”成一维的纯文本字符流，是当时唯一可行、成本最低的输入和存储方式。

## 3. 它支撑了什么理论、工具或制度？（影响面）
基于“代码即文本”的预设，人类构建了整个现代软件工程的基础设施：
* 我们用 **Git** 来做版本控制，它的底层逻辑是“对比两行文本的差异（Diff）”。
* 我们用 **Linter** 和 **Prettier** 来做代码格式化。
* 我们用正则表达式去全局搜索代码。
* 甚至现在的 AI 编程（如 Copilot），本质上也是在玩“预测下一个文本字符”的游戏。

## 4. 哪些异常长期解释不了？（现实冲突）
因为这个地基是错的，全球几千万程序员每天都在经历荒谬的痛苦：
* **荒谬的 Git 冲突**：程序员 A 把变量名改了，程序员 B 在下一行加了一个空格。在逻辑上，这两人完全没有干涉。但 Git 是基于“文本行”对比的，它立刻大报警：“严重冲突！请手动解决！”
* **无意义的圣战**：为了缩进到底用 Tab 还是 4 个空格，大括号要不要换行，程序员吵了 30 年。

## 5. 当时做了哪些局部修补？（补丁坍塌）
为了弥补这些问题，我们发明了极其复杂的 Git 分支管理流、配置了成百上千条规则的格式化工具、在 CI/CD 里加入严格的语法检查……**整个行业每年浪费上千亿美元，仅仅是为了解决“文本对不齐”的问题。**

## 6. 谁把这个前提显性化了？
现代的编译器设计者和一些前沿的结构化编辑器（如 JetBrains MPS, Unison 语言）指出：**计算机根本不认识文本！** 编译器拿到文本的第一秒，就是把它转换成多维的“抽象语法树（AST / 逻辑图谱）”。
真相是：**代码的“内容本体”是那棵逻辑树（后端数据），而“纯文本”仅仅是为了让人类肉眼能看懂而强行降维渲染出来的“前端 UI”！**

## 7. 前提改写后打开了什么新空间？
一旦我们将这个隐性前提显性化，并重构地基（直接将代码作为逻辑图谱存入数据库）：
1. **格式化圣战终结**：文本只是 UI。你想看 Tab，你的屏幕就渲染成 Tab；我想看 4 个空格，我的屏幕就渲染成空格。底层逻辑数据根本没变！
2. **Git 冲突消失 90%**：只要逻辑节点没有交叉，无论文本怎么换行，系统都能自动完美合并。
3. **语法错误成为历史**：你直接操作的是逻辑节点，再也不会出现“漏写一个右括号”这种低级的文本拼写错误。

## 8. 总结与启示
我们为了 1970 年代显示器的“输出效率”做出了妥协（把逻辑降维成文本），却在随后的几十年里几乎遗忘了这本是一个妥协。我们在这个“文本地基”上不断进行局部修复。这很好地展示了**“把 UI 投影（地图）误认为数据本体（领土）”**的代价。

## 9. 条件与边界分析（v2：探针输出升级补充）

* **① 这个假设成立的条件是什么？**
  早期是“终端与存储只能处理一维 ASCII 字符流”这个硬件约束；但它能延续至今，靠的是一个**至今依然真实成立的巨大优势**——文本极其通用、可移植、人类可读、跨语言跨工具，几乎是所有系统的“最低公约数”。
* **② 这个条件今天还在不在？**
  硬件约束**早已消失**；但“文本是通用最低公约数”这个**优势依然非常强**。这是本案例必须诚实承认的一点——**这正是为什么不能说“行业错了 50 年”**。文本不是愚蠢，而是一个极其成功的通用妥协。
* **③ 在哪些新场景里，这个条件已经悄悄消失/松动了？**
  在**对“语义精度”要求极高的新场景**里，纯文本的边界开始暴露：结构化编辑器（Scratch、Blockly）、语义级 Diff（tree-sitter、GumTree）、语义合并、内容寻址的 Unison 语言，以及 **AI 编程**——大模型其实正在被迫重新理解代码的“结构”，而不只是“字符”。

## 证据等级与表达校准（按《证据闸门》《证伪协议》）

* **证据等级：B 级。**
* **校准说明**：早期版本里“**这个地基是错的**”“**整个行业每年浪费上千亿美元**”等说法**过于绝对，且数字无来源**，已在正文中弱化。文本化是一个在通用性上极其成功的妥协，只是在 AI 编程与超大规模协作时代开始显出边界。
* **对外可以说**：文本是“成功的通用妥协”，但在 AI 时代，把“文本差异”等同于“语义差异”的代价正在变大。
* **对外不能说太满**：不要说“行业错了 50 年”“Git 是错的”“冲突能消失 90%”这类无证据的断言。
