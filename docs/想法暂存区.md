# 想法暂存区（Backlog）

> **这份文档的唯一规矩：捕捉 ≠ 承诺。**
> 把想法写在这里，不代表要去做它。它只是一个让你"敢把想法放下、不怕丢"的地方。

## A. 可以做成原型的（候选，最多挑一个动手）

### A1. 假设挖掘机（递归前提提取器）
* **一句话**：拿到任何观点/论据，机械地问"它底下垫着什么假设？"，挖出来，再对每个假设递归往下问，直到最底层。
* **本质**：它是手记里"儿童般连环追问"的机器版；机器不会因"这还用问吗"而停下，所以能穿透"司空见惯"。
* **定位**：它是机器流水线缺失的"第一道工序"——
  `① 假设挖掘机(挖出假设树) → ② 探针(标出可疑节点) → ③ 证据闸门(滤掉良性的) → ④ 条件与边界(刻画幸存者)`

* **设计修正：不是"剪枝删除"，而是"分级加工 + 排序"**：
  初版设想"只挖可疑节点、其余剪掉"是危险的——因为问题假设可能埋在第 10000 层，且可能在**中间**而非底层；浅层看着无聊的分支，深处可能正藏着裂缝。**砍掉 = 不可逆地丢信息（假阴性）。**
  正确做法是做成**多级加工流水线，只排序、不删除**：
  - **初加工**：广度优先、浅层快速扫面，给所有分支打"可疑度"并排序（不删任何分支，只排队）。
  - **正加工**：可疑度高的分支优先往深挖；低的留在队列里，不丢。
  - **深加工**：对幸存分支挖到底。
  - 探针在**每一层节点**都要跑（问题可能在任意深度，不只在叶子）。
  - 口诀：**砍掉是赌博，排序是策略。**

* **为什么非深挖不可（规模定律）**：你不挖到底，就不知道哪一层有问题。人因资源受限（专注约 35-40 瓦）往往挖几层就放弃——这正是历史上"挖地基"极其罕见的根本原因。**机器的真正价值，就是完成"人类因资源受限而无法完成的深度搜索"。** 规模定律不只适用于大模型训练，也适用于人类知识的积累（几千年、无数人的叠加）。机器在这里不是替人聪明，而是替人"不知疲倦地往下挖"。

* **诚实的残留难点**：分级加工只能**管理**爆炸、**提高**撞见概率，并不能**消除**爆炸或**保证**找到深层裂缝。若裂缝藏在一条浅层毫无征兆的分支的极深处，任何优先级启发式都可能错过——只能靠机器的规模尽量逼近穷尽。

* **状态**：今晚最具操作性、最值得做第一个真实原型的候选。

## B. 待挖的研究线索（先park，不急）

* **B1. 论文作为"突入点"**：论文知识密度最高，且为了过审会省略大量"理所当然"的前提——而被省略的，正是地基。注意区分两类省略：领域共识型（良性，别挖）vs 从未被审查的深层预设（目标）。
* **B2. 跨领域对撞**：单篇论文的隐性前提常是全圈共享的盲区，连作者都看不见。把 A 领域"天经地义"的前提丢到 B 领域，往往一撞就现形。可能是论文路线里最关键的一招。
* **B3. 异常为何被"视而不见"**：长期暴露 → 习惯化 → 盲视。如何系统对抗"习惯化盲视"？（已在 `项目定位与目标.md` 列为线索）
* **B4. 神经元放电 vs 意识**：用"神经元只是放电"去否定高层心智，是否也是一次"基底决定论"式的纵向层级混淆？（见 `cases/05_Transformer_VM.md` 结尾）
* **B5. "组织关系"能否被有限地描述**：承认"整体 = 部分 + 关系"后，关系是"有限但太大"（纯资源问题），还是"动态生成、边写边变"（对象本身在演化）？（见 `07_孪生陷阱_整体与部分.md`）
